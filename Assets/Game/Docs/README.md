# 기능 명세 (Docs)

`Assets/Game/Docs/` 아래에 **기능별·도메인별** 구현 메모와 **추후 확인 사항(Future)** 을 둡니다.

## 목적

- 지금 당장 구현하지 않지만, 나중에 손댈 때 놓치기 쉬운 제약·체크리스트를 남김
- 해당 기능 작업 시 AI·개발자가 **같은 문서를 다시 읽고** README·채팅에 한 번 더 언급

## 문서 목록

| 문서 | 범위 |
|------|------|
| [Movement.md](Movement.md) | 이동·점프·갈고리 |
| [PlayerAnimation.md](PlayerAnimation.md) | Visual·Animator·콤보 공격 애니 |
| [Combat.md](Combat.md) | ComboDefinition·히트박스 SO·이벤트 |

## 작성 규칙

1. **파일명** — `PascalCase` 또는 도메인명 (`PlayerAnimation.md`)
2. **상단** — 현재 구현 상태 표 (✅ / ⏸ / ❌)
3. **「추후 구현 시 확인 (Future)」** — 체크리스트 (`- [ ]`) 로 구체적 항목
4. **변경 이력** — 날짜·요약 한 줄
5. **README.md** `## 문서` 에 링크 추가

## 워크플로 (기능 작업 시)

1. README `## 문서` 에서 관련 md 링크 확인
2. 해당 md의 **Future** 체크리스트 읽기
3. 구현·튜닝 후 체크리스트·상태 표·변경 이력 갱신

## 주의 (한계)

| 리스크 | 대응 |
|--------|------|
| 문서와 코드 불일치 | 기능 PR·작업 마무리 시 md 갱신을 습관화 |
| Future만 쌓이고 안 지움 | 완료한 항목은 체크하거나 본문으로 옮기고 삭제 |
| 너무 거대한 한 파일 | 도메인별로 쪼개기 (Movement / PlayerAnimation 등) |

**결론:** md 명세 방식은 문제 없음. **링크 유지 + 작업 끝에 갱신**만 지키면 됩니다.
